# REQ-002: Tracker Setup & Execution Requirements

## 1. Objective

Phase 2 converts confirmed Discovery outputs into a durable Visibility Tracker, generates and confirms prompts, configures platforms/cadence, executes scan runs, captures AI answers, streams progress, and hands off completed/partial scan runs to Phase 3 Analysis.

## 2. Scope

In scope:

```text
Visibility Tracker creation
Suggested tracker name
Backend coverage creation
Prompt generation
Prompt review
Custom prompt addition
Prompt regeneration
Prompt confirmation
Platform selection
Cadence selection
Tracker edit rules
ScanRun creation
PromptRun creation
Background execution
AI platform reference data
Provider adapter execution
Live progress via SSE/WebSocket
AIAnswer storage
Raw provider response storage
Retries
Cancellation
Partial completion
Scheduled scan creation
Scheduled scan email notifications
AnalysisJob auto-trigger
Operational/event hooks
```

Out of scope:

```text
Analysis logic
Finding generation details
Content Action generation details
PDF report design
Pricing/plan enforcement
CMS/social publishing
Focus Area workflows
Per-workspace platform configuration
Prompt editing UX
```

## 3. Functional requirements

### FR-001: Create Visibility Tracker

After Discovery confirmation, the system shall show a “ready to create tracker” screen with:

```text
Suggested tracker name
Summary of confirmed brand details
Summary of topics, competitors, products/services, market
Explanation of what the tracker will do
Primary CTA: Create Visibility Tracker
```

The suggested name shall be editable.

### FR-002: Create TrackerConfiguration

On create, the backend shall create `TrackerConfiguration` with:

```text
BrandId
Name
PromptAllocation
Selected platforms/models
Cadence, initially unset or defaulted later
Status = Draft until prompts/platform/cadence confirmed
```

### FR-003: Create backend Coverage

The backend shall create tracker coverage from confirmed Discovery outputs and default Visibility Checks.

Coverage is not a user-facing step.

### FR-004: Generate draft prompts

The system shall generate Draft prompts using:

```text
BrandProfile
Category
Products/services
Topics
Competitors
Market
Audience optional
Visibility Checks
Prompt templates
PromptAllocation
```

### FR-005: Prompt review

The user shall review generated Draft prompts before the first scan.

Allowed actions:

```text
Remove prompt
Add custom prompt
Regenerate all prompts
Regenerate by Topic
Regenerate by Visibility Check
Regenerate by Topic + Visibility Check
Confirm prompts
```

Prompt text editing is not supported in v1.

### FR-006: Custom prompt mappings

When a user adds a custom prompt, the system shall infer mappings and show suggested values.

Suggested mappings:

```text
Topic
Visibility Check
Competitor
Product / Service
Audience
Market
```

High-confidence mappings shall be preselected. The user may edit suggestions.

### FR-007: Confirm prompts

When user confirms prompts:

```text
Draft prompts become Active.
Only Active prompts are used in scan runs.
```

### FR-008: Platform selection

After prompt confirmation, the user selects platforms/models.

V1 platforms:

```text
ChatGPT
ChatGPT Search
Gemini
Claude
```

Default: all selected.

Selected platforms/models are fixed after tracker creation.

### FR-009: Cadence selection

After prompt confirmation, the user selects cadence:

```text
On demand
Daily
Weekly
Monthly
```

Default: Weekly.

Cadence can be changed after tracker creation.

### FR-010: Run first scan

When user clicks Run scan:

```text
Create ScanRun
Create PromptRuns for Active Prompt × selected AIPlatform
Queue PromptRuns as background jobs
Return ScanRunId immediately
Open progress screen
```

### FR-011: Live progress

The UI shall display live progress through SSE/WebSocket.

Progress unit:

```text
Scan Check = Prompt × AI Platform
```

Progress copy:

```text
36 / 120 scan checks complete
```

Polling fallback should exist.

### FR-012: Background execution

PromptRuns shall execute independently in parallel, controlled by platform-specific concurrency limits.

### FR-013: Retry failures

PromptRuns shall retry transient provider failures up to 2 times with backoff.

### FR-014: Store AI answers

For each completed PromptRun:

```text
Store clean answer text in DB.
Store full raw provider response in object storage.
Store raw response URL and metadata on AIAnswer.
```

### FR-015: Cancel scan

User may cancel a running scan.

Cancellation behavior:

```text
Queued PromptRuns cancelled.
No new PromptRuns start.
Completed answers remain stored.
ScanRun marked Cancelled.
No final findings generated by default.
```

### FR-016: Partial completion

If at least one PromptRun completes and at least one fails after retries, mark ScanRun `PartiallyCompleted`.

Partial scans shall still trigger analysis and generate findings based on available answers.

Results must be clearly labelled partial.

### FR-017: Store tracker snapshot

Every ScanRun shall store `TrackerSnapshotJson` containing the tracker setup used at run time.

### FR-018: Scheduled scans

If tracker is Active and cadence is Daily/Weekly/Monthly and NextRunAt is due, the system shall create a scheduled ScanRun.

Minimum scheduled cadence is Daily.

No complex overlap handling is required in v1. If a scan is already running for the same tracker, skip and log.

### FR-019: Scheduled scan notifications

Scheduled scans shall automatically generate findings/content actions and send an email summary to configured recipients.

PDF reports shall not auto-generate in v1.

### FR-020: Notification settings

Tracker shall support notification settings and recipients.

Email summary includes:

```text
Tracker name
Scan date
Scan status
Top findings
Top content actions
Link to web results
Partial/failure warning if applicable
```

### FR-021: Analysis handoff

When ScanRun reaches Completed or PartiallyCompleted, automatically enqueue AnalysisJob.

Failed or Cancelled scans do not auto-trigger analysis.

## 4. Non-functional requirements

```text
Scan execution must not block web requests.
Progress updates should be near real-time.
System should support concurrent scans with provider throttling.
Raw provider responses must be retained for audit/reprocessing.
Historical scan runs must remain explainable after tracker edits.
Errors/failures must be visible to operations and summarized to users.
```

## 6. Acceptance criteria

```text
User can create a Visibility Tracker after Discovery.
System generates draft prompts.
User can review, remove, add custom, regenerate, and confirm prompts.
User can select platforms and cadence after prompt confirmation.
User can run first scan.
Scan progress updates live using scan checks.
Scan execution runs in background.
AI answers and raw responses are stored.
Partial scans are supported and labelled.
Scheduled scans create scan runs and email summaries.
Completed/partial scan runs automatically trigger analysis jobs.
```
